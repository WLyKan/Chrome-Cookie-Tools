# 扩展版本更新提醒实现计划

## 目标
在不改动现有核心读写流程的前提下，为 popup 增加“启动即检查更新”能力：对比当前版本与 GitHub 最新稳定 tag，发现新版本时展示提示框。

## 关键改动点
- 在 [D:/Codes/chrome-cookie-tools/src/entrypoints/background.ts](D:/Codes/chrome-cookie-tools/src/entrypoints/background.ts) 新增“检查 GitHub 最新版本”消息处理分支（统一走现有 `onMessage` 路由）。
- 在 [D:/Codes/chrome-cookie-tools/src/types.ts](D:/Codes/chrome-cookie-tools/src/types.ts) 扩展消息类型与响应类型，定义更新检查请求/响应结构（含当前版本、最新版本、是否有更新、发布链接、错误信息）。
- 在 [D:/Codes/chrome-cookie-tools/src/entrypoints/popup/App.tsx](D:/Codes/chrome-cookie-tools/src/entrypoints/popup/App.tsx) 新增首次加载检查逻辑：调用 background 消息并触发弹窗。
- 复用 [D:/Codes/chrome-cookie-tools/package.json](D:/Codes/chrome-cookie-tools/package.json) 当前版本（运行时通过 `browser.runtime.getManifest().version` 读取，不手写硬编码版本，避免漂移）。

## 实现步骤
1. **类型层**：新增 `MessageType.CHECK_EXTENSION_UPDATE`、请求/响应类型与版本比较结果模型，保证 TypeScript 类型安全。
2. **后台层**：
   - 解析当前扩展仓库 remote（owner/repo）。
   - 请求 GitHub Releases API（只取稳定版，过滤 `prerelease` / `draft`）。
   - 规范化 tag（处理 `v1.2.3`）并做 semver 对比。
   - 返回结构化结果；网络失败/解析失败时返回可读错误信息。
3. **Popup 层**：
   - `App` 启动时触发一次检查（带防抖与短期缓存，避免每次切 tab 重复请求）。
   - 若有更新，弹出提示框（含最新版本号 + “去下载/查看发布页”）。
   - 无更新或失败时静默/轻提示，不影响主流程。
4. **持久化策略**：在 `browser.storage.local` 保存最近一次检查时间和最近提醒版本，避免短时间重复打扰。
5. **验证**：
   - 本地模拟“当前版本低于 tag”与“已是最新版”两条路径。
   - 校验离线/接口异常时的容错提示。
   - 运行 `pnpm compile` 确认类型通过。

## 设计约束
- 不引入新依赖，使用现有运行时与类型体系。
- 不改动现有 storage 读写核心逻辑，仅新增独立更新检查链路。
- 提示框默认只在发现新稳定版时出现，避免噪音。
